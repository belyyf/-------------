Описание проекта по дисциплинам
«Технология разработки программного обеспечения» и
«Инструментальные средства разработки программного обеспечения»
Общая информация
В процессе выполнения проекта вы возьмете на себя задачу управления полноценным проектам, аналоги которых присутствуют на производстве в IT-компаниях.
Конечно, проект учебный, и потому слишком сложное приложение создавать не нужно. Главное — понять принцип на уровне ознакомления.
Идея
Ваша цель — произвести разработку определенного программного продукта в соответствии со стандартной технологией разработки программного обеспечения и с использованием средств разработки.
Задачи проекта перечислены в соответствующем пункте документа.

Необходимо уточнить несколько важных требований:
•	Реализация программного продукта производится в команде из (оптимально) 3 человек.
•	Каждый член команды примеряет на себя от двух до четырех ролей (должностей) и выполняет поставленные задачи строго по ним. Каждый член команды должен побыть «программистом».
•	Программный продукт может быть написан на любом языке программирования с использованием любых библиотек и фреймворков.
•	Программный продукт должен иметь модульную архитектуру (хотя бы два заметных компонента), предпочтительно использование парадигмы ООП и архитектурных шаблонов.
•	При разработке должны быть обязательно использованы средства, о которых рассказывается на дисциплине «Инструментальные средства разработки программного обеспечения».
•	При разработке должны быть обязательно подготовлены материалы по темам дисциплины «Технология разработки программного обеспечения».
Форма отчетности
В результате выполнение проекта должны быть сформированы:
•	Удаленный репозиторий на GitHub с понятной структурой, наличием документации и процессом развертывания. Данная часть отчетности по заданию включает последовательное применение средств, связанных с выполнением каждой задачи ИСРПО.
•	Качественно оформленный отчетный документ с описанием выполнения каждой задачи ТРПО (если нужен шаблон, об этом стоит сообщить заранее, при оформлении можно опираться на «ГОСТ 7.32-2017. СИБИД. Отчет о научно-исследовательской работе»).
Задачи ТРПО
Задача 1. Инициация проекта.
Сформируйте команду из нескольких человек, определитесь с темой проекта, распределите роли между участниками (хотя бы приблизительно), сформулируйте бизнес-требования, будто их вам отдает заказчик. Возможные роли участников:
- Менеджер проекта (обязательно)
- Аналитик (обязательно)
- Архитектор (обязательно)
- Программист (обязательно все)
- Тестировщик (обязательно)
- DevOps-инженер (обязательно)
- Технический писатель (обязательно)
- UX/UI-дизайнер (необязательно: если предполагается графический интерфейс)
- Проектировщик БД (необязательно: если в программе присутствует работа с БД)
и другие.
Учтите, что проект должен быть небольшой, если есть желание реализовать собственную задумку. Составьте техническое задание, за основу можно взять документ «ГОСТ 19.201-78. ЕСПД. Техническое задание».

Задача 2. Определение методологии разработки.
Согласно идее проекта подберите наиболее подходящую методологию под ваш проект. Объясните свой выбор. В соответствии с методологией определите развитие жизненного цикла продукта.

Задача 3. Управление проектом.
Данная задача возлагается на участника, исполняющего роль менеджера проекта. Для каждого бизнес-требования создайте задачи, назначьте их на участников с нужной ролью, оцените время выполнения каждой задачи. Учтите последовательность выполнения задач согласно вашей выбранной методологии.
В процессе выполнения проекта данная задача будет присутствовать постоянно и периодически обновляться из-за движения по жизненному циклу продукта. Менеджер проекта должен контролировать выполнение задач участников команды в нужные сроки.

Задача 4. Аналитика.
Данная задача возлагается на участника, исполняющего роль аналитика. Составьте функциональные и нефункциональные требования продукта. Проанализируйте возможные сценарии действия пользователя.

Задача 5. Архитектура.
Данная задача возлагается на участника, исполняющего роль архитектора (проектировщика, дизайнера). Выполните проектирование модели приложения с использованием диаграмм UML (диаграммы компонентов, классов, вариантов использования). В случае наличия графического интерфейса изобразите схему интерфейса и диаграмму состояний. В случае наличия БД изобразите диаграмму сущность-связь. Опишите техническое решение для каждого модуля программы, включите в описание созданные схемы.

Задача 6. Разработка.
Данная задача возлагается на участников, исполняющих роли программистов. Выполните реализацию модулей согласно представленному техническому решению. Разработка должна подкрепляться частыми фиксациями и выполняться параллельно для нескольких модулей. Слияние различного функционала в одну ветвь должно происходить с использованием механизма Pull Requests.

Задача 7. Тестирование.
Данная задача возлагается на участников, исполняющих роли программистов и тестировщиков. Для программных модулей напишите ряд модульных тестов (unit-tests), убедитесь, что тестирование проходит успешно. Для всей системы целиком составьте несколько тестовых сценариев, проверяющих функциональность программы согласно требованиям. Выполните ручное тестирование, интеграционное тестирование. При обнаружении дефектов публикуйте его описание в GitHub Issues.

Задача 8. Интеграция и сопровождение.
Данная задача возлагается на участников, исполняющих роли программистов, DevOps-инженера и технического писателя. Подготовьте продукт к развертыванию, настройте GitHub Actions для выполнения сборки с запуском автоматического тестирования. Подготовьте документацию кода (комментарии Doxygen). Подготовьте инструкцию для использования продукта.

Дополнительная задача: опубликуйте релизную версию продукта в виде артефакта в GitHub Releases, создайте тег.
Задачи ИСРПО
Задача 1. Создание и настройка репозитория.
Создайте репозиторий на GitHub для вашего продукта. Репозиторий должен иметь публичную видимость. Пригласите участников команды в ваш репозиторий. Настройте репозиторий следующим образом (используйте Rulesets):
- основная ветка main;
- прямые фиксации в основную ветку запрещены;
- слияние с основной веткой происходит с использованием Pull Requests;
- Pull Requests должен подтвердить хотя бы один участник, Pull Request нельзя подтвердить для самого себя, все комментарии перед слиянием должны быть разрешены (resolved).
Опубликуйте README файл с описанием идеи вашего продукта.

Задание 2. Управление проектом.
Согласуется с задачей 3 по ТРПО. Используя GitHub Projects сформируйте набор задач для каждой роли проекта. Определите приоритет и тип каждой задачи, исполнителя и время в часах на ее выполнение.

Задача 3. Развертывание — контейнеризация.
Согласуется с задачей 8 по ТРПО. Сформируйте собственный образ Docker на основе необходимого набора пакетов для сборки вашего продукта. Учтите особенности команд Dockerfile при скачивании зависимостей.

Задача 4. Развертывание — сборка.
Согласуется с задачей 8 по ТРПО. Настройте шаги GitHub Actions для сборки вашего проекта в Docker с использованием созданного контейнера. Сборка проекта должна запускать модульные тесты, написанные под ваш продукт.

Задача 5. Ведение документов.
Согласуется с задачей 8 по ТРПО. Составьте документы, описывающие предпринятые решения по продукту в сервисе GitHub Wikis. Внутри должно быть представлено подробное описание структуры программы, требования, технические решения по разным модулям, тестовые сценарии, должен быть описан процесс развертывания.

Задача 6. Документирование кода.
Согласуется с задачами 6 и 8 по ТРПО. Выполните генерацию документации исходного кода в виде страниц HTML. Часть документации можно перенести в GitHub Wikis. При желании опубликуйте документацию исходного кода на сайте GitHub Pages.
